2.5 Snapshot Strategy Recommendation

Populate 4 columns at screening creation time to eliminate 6 of the 13 API calls above: candidate_name_snapshot (already exists on ai_screenings), candidate_email_snapshot (add), job_title_snapshot (already exists), job_description_snapshot (add). These are already partially present in the model. Filling them at creation reduces the Screening Service's runtime dependency on the Monolith to 3 callsites: question generation (needs live resume data), candidate Q&A (needs live job description), and pipeline advancement (write).

3.3 Migration 002 Amendment Required

The blueprint's Migration 002 is missing a critical DROP. Before ALTER TABLE public.profiles SET SCHEMA auth can succeed, the inbound FK from public.jobs must be dropped first. Add this to the top of Migration 002:

-- ADD to top of 002_move_auth_tables.sql (BEFORE the SET SCHEMA statements)

-- public.jobs has a FK pointing AT profiles. PostgreSQL will refuse to move
-- profiles to a different schema while this inbound FK exists.
-- We drop it here and enforce it at the application layer.
ALTER TABLE public.jobs
    DROP CONSTRAINT IF EXISTS fk_jobs_created_by_profiles;

-- After this, proceed with the SET SCHEMA statements as written in Migration 002.

3.4 Supabase Schema Name Collision Warning

Verify before running: Supabase uses a built-in schema named auth for its own authentication tables (including auth.users). If you CREATE SCHEMA auth on a Supabase project, it will either fail (schema already exists) or silently become the same namespace — mixing your application tables with Supabase internals. Recommended: rename your target schema to app_auth in all migration scripts and all SQLAlchemy model __table_args__. Update Migrations 001–005 and Section 6.1 of the blueprint accordingly.

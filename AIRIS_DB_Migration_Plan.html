Phase 3 Compatibility Strategy

3.1 — Strategy Overview

Zero-downtime is achieved by creating compatibility views in public that mirror every moved table. The ORM continues resolving ForeignKey("profiles.id") to public.profiles — which the view redirects transparently. During Phase 1A/1B, the application reads/writes through the views unchanged. Phase 1C removes the views once all ORM model schema tags have been applied.

PhaseActionApp behavior
Phase 1ACreate schemas + Move tables + Create compat viewsApp is unchanged — reads/writes via views, ORM models unchanged
Phase 1BApply schema tags to ORM models (__table_args__ = {"schema": "app_auth"}), update ForeignKey refsApp reads/writes directly to new schemas; views still exist as fallback
Phase 1CDrop compat views, drop old public FK stubs, seed new alembic_version tablesApp runs fully on separated schemas; no views

3.4 — RLS Policy Updates

No RLS policies were found in application migrations. However, Supabase may have auto-generated policies on auth tables. Before executing: run SELECT schemaname, tablename, policyname, cmd, qual FROM pg_policies WHERE schemaname = 'public' ORDER BY tablename against production. Any policies on tables being moved must be recreated on the new schema after SET SCHEMA (policies are NOT automatically moved). Use the pattern below.
-- Template: recreate any RLS policies found on moved tables
-- Run BEFORE the migration to capture existing policies:
SELECT
  tablename,
  policyname,
  cmd,
  qual,
  with_check
FROM pg_policies
WHERE schemaname = 'public'
  AND tablename IN (
    'organizations', 'profiles', 'auth_sessions', 'organization_roles',
    'permissions', 'role_permissions', 'invites',
    'ai_screenings', 'ai_screening_questions', 'ai_screening_answers',
    'ai_screening_evaluations', 'ai_screening_messages', 'ai_screening_segments'
  );

ORDER BY tablename;

FROM pg_class
WHERE relnamespace = to_regnamespace('public')
  AND relkind = 'r'
ORDER BY relname;

-- 0c. Capture ALL foreign key constraint names (critical — use these in Step 2)
SELECT
  tc.constraint_name,
  tc.table_name AS from_table,
  kcu.column_name AS from_col,
  ccu.table_name AS to_table,
  ccu.column_name AS to_col,
  rc.delete_rule
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu
  ON tc.constraint_name = kcu.constraint_name
JOIN information_schema.constraint_column_usage ccu
  ON tc.constraint_name = ccu.constraint_name
JOIN information_schema.referential_constraints rc
  ON tc.constraint_name = rc.constraint_name
WHERE tc.constraint_type = 'FOREIGN KEY'
  AND tc.table_schema = 'public'
ORDER BY tc.table_name, kcu.column_name;

SELECT tablename, policyname, cmd, qual, with_check

WHERE schemaname = 'public'
ORDER BY tablename;

1. Scope & Constraints

In Scope — Phase 1

  • Extract Auth Service → auth schema
  • Extract AI Screening Service → screening schema
  • Remaining ATS monolith stays in public schema
  • All cross-service data access via REST API calls only

Hard Constraints

  • Single PostgreSQL instance — no new DB servers
  • Schema-based isolation: auth / screening / public
  • No direct cross-schema SQL from application code
  • Supabase Storage remains shared — no change

Migration 002 — Move Auth Tables + Compatibility Views

-- 002_move_auth_tables.sql
-- Compatibility views keep the monolith running with ZERO code changes
-- during the transition period. Drop them only in Phase 1C once all
-- cross-schema SQL has been replaced by API calls.

BEGIN;

ALTER TABLE public.profiles           SET SCHEMA auth;
ALTER TABLE public.organizations      SET SCHEMA auth;
ALTER TABLE public.auth_sessions       SET SCHEMA auth;
ALTER TABLE public.organization_roles  SET SCHEMA auth;
ALTER TABLE public.permissions         SET SCHEMA auth;
ALTER TABLE public.role_permissions    SET SCHEMA auth;
ALTER TABLE public.invites             SET SCHEMA auth;

-- Compatibility views so existing monolith queries keep working transparently
CREATE OR REPLACE VIEW public.profiles           AS SELECT * FROM auth.profiles;
CREATE OR REPLACE VIEW public.organizations      AS SELECT * FROM auth.organizations;
CREATE OR REPLACE VIEW public.auth_sessions       AS SELECT * FROM auth.auth_sessions;
CREATE OR REPLACE VIEW public.organization_roles  AS SELECT * FROM auth.organization_roles;
CREATE OR REPLACE VIEW public.permissions         AS SELECT * FROM auth.permissions;
CREATE OR REPLACE VIEW public.role_permissions    AS SELECT * FROM auth.role_permissions;
CREATE OR REPLACE VIEW public.invites             AS SELECT * FROM auth.invites;

COMMIT;

Migration 003 — Move AI Screening Tables + Compatibility Views

-- 003_move_screening_tables.sql
-- PostgreSQL cannot enforce FK constraints across schemas.
-- All cross-schema FKs must be dropped before SET SCHEMA,
-- then recreated as intra-schema FKs within screening.

BEGIN;

-- 1. Drop cross-schema FK constraints (adjust names to match your Alembic output)
ALTER TABLE public.ai_screenings
    DROP CONSTRAINT IF EXISTS ai_screenings_candidate_id_fkey,
    DROP CONSTRAINT IF EXISTS ai_screenings_job_id_fkey,
    DROP CONSTRAINT IF EXISTS ai_screenings_pipeline_id_fkey,
    DROP CONSTRAINT IF EXISTS ai_screenings_organization_id_fkey;

ALTER TABLE public.ai_screening_questions
    DROP CONSTRAINT IF EXISTS ai_screening_questions_screening_id_fkey;

ALTER TABLE public.ai_screening_answers
    DROP CONSTRAINT IF EXISTS ai_screening_answers_screening_id_fkey,
    DROP CONSTRAINT IF EXISTS ai_screening_answers_question_id_fkey;

ALTER TABLE public.ai_screening_evaluations
    DROP CONSTRAINT IF EXISTS ai_screening_evaluations_screening_id_fkey,
    DROP CONSTRAINT IF EXISTS ai_screening_evaluations_question_id_fkey;

ALTER TABLE public.ai_screening_messages
    DROP CONSTRAINT IF EXISTS ai_screening_messages_screening_id_fkey;

ALTER TABLE public.ai_screening_segments
    DROP CONSTRAINT IF EXISTS ai_screening_segments_screening_id_fkey;

-- 2. Move tables into the screening schema
ALTER TABLE public.ai_screenings            SET SCHEMA screening;
ALTER TABLE public.ai_screening_questions   SET SCHEMA screening;
ALTER TABLE public.ai_screening_answers     SET SCHEMA screening;
ALTER TABLE public.ai_screening_evaluations SET SCHEMA screening;
ALTER TABLE public.ai_screening_messages    SET SCHEMA screening;
ALTER TABLE public.ai_screening_segments    SET SCHEMA screening;

-- 3. Restore intra-schema FKs (all within the screening schema)
ALTER TABLE screening.ai_screening_questions
    ADD CONSTRAINT ai_screening_questions_screening_id_fkey
    FOREIGN KEY (screening_id)
    REFERENCES screening.ai_screenings(id) ON DELETE CASCADE;

ALTER TABLE screening.ai_screening_answers
    ADD CONSTRAINT ai_screening_answers_screening_id_fkey
    FOREIGN KEY (screening_id)
    REFERENCES screening.ai_screenings(id) ON DELETE CASCADE,
    ADD CONSTRAINT ai_screening_answers_question_id_fkey
    FOREIGN KEY (question_id)
    REFERENCES screening.ai_screening_questions(id) ON DELETE SET NULL;

ALTER TABLE screening.ai_screening_evaluations
    ADD CONSTRAINT ai_screening_evaluations_screening_id_fkey
    FOREIGN KEY (screening_id)
    REFERENCES screening.ai_screenings(id) ON DELETE CASCADE,
    ADD CONSTRAINT ai_screening_evaluations_question_id_fkey
    FOREIGN KEY (question_id)
    REFERENCES screening.ai_screening_questions(id) ON DELETE CASCADE;

ALTER TABLE screening.ai_screening_messages
    ADD CONSTRAINT ai_screening_messages_screening_id_fkey
    FOREIGN KEY (screening_id)
    REFERENCES screening.ai_screenings(id) ON DELETE CASCADE;

ALTER TABLE screening.ai_screening_segments
    ADD CONSTRAINT ai_screening_segments_screening_id_fkey
    FOREIGN KEY (screening_id)
    REFERENCES screening.ai_screenings(id) ON DELETE CASCADE;

-- NOTE: ai_screenings.candidate_id / .job_id / .pipeline_id are intentionally
-- left as unforced UUID columns. The Screening Service validates existence
-- via API calls to the Monolith before creating a screening record.

-- 4. Compatibility views so the monolith keeps working without code changes
CREATE OR REPLACE VIEW public.ai_screenings
    AS SELECT * FROM screening.ai_screenings;
CREATE OR REPLACE VIEW public.ai_screening_questions
    AS SELECT * FROM screening.ai_screening_questions;
CREATE OR REPLACE VIEW public.ai_screening_answers
    AS SELECT * FROM screening.ai_screening_answers;
CREATE OR REPLACE VIEW public.ai_screening_evaluations
    AS SELECT * FROM screening.ai_screening_evaluations;
CREATE OR REPLACE VIEW public.ai_screening_messages
    AS SELECT * FROM screening.ai_screening_messages;
CREATE OR REPLACE VIEW public.ai_screening_segments
    AS SELECT * FROM screening.ai_screening_segments;

COMMIT;

Migration 004 — Database Role Permissions per Service

-- 004_db_roles.sql  (run as superuser)
-- On Supabase: manage connection strings per service via the Dashboard
-- rather than creating Postgres roles here.

DO $$ BEGIN
    IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname = 'airis_auth')
        THEN CREATE ROLE airis_auth LOGIN PASSWORD 'replace_me'; END IF;
    IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname = 'airis_screening')
        THEN CREATE ROLE airis_screening LOGIN PASSWORD 'replace_me'; END IF;
    IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname = 'airis_monolith')
        THEN CREATE ROLE airis_monolith LOGIN PASSWORD 'replace_me'; END IF;
END $$;

-- Auth Service: full access to auth schema only
GRANT USAGE ON SCHEMA auth TO airis_auth;
GRANT ALL PRIVILEGES ON ALL TABLES    IN SCHEMA auth TO airis_auth;
GRANT ALL PRIVILEGES ON ALL SEQUENCES IN SCHEMA auth TO airis_auth;
ALTER DEFAULT PRIVILEGES IN SCHEMA auth
    GRANT ALL ON TABLES TO airis_auth;

-- Screening Service: full access to screening schema only
GRANT USAGE ON SCHEMA screening TO airis_screening;
GRANT ALL PRIVILEGES ON ALL TABLES    IN SCHEMA screening TO airis_screening;
GRANT ALL PRIVILEGES ON ALL SEQUENCES IN SCHEMA screening TO airis_screening;
ALTER DEFAULT PRIVILEGES IN SCHEMA screening
    GRANT ALL ON TABLES TO airis_screening;

-- Monolith: full access to public; READ-ONLY to auth/screening (Phase 1A/B only)
-- REVOKE the auth/screening grants in Phase 1C once compatibility views are dropped.
GRANT USAGE ON SCHEMA public, auth, screening TO airis_monolith;
GRANT ALL PRIVILEGES ON ALL TABLES    IN SCHEMA public    TO airis_monolith;
GRANT ALL PRIVILEGES ON ALL SEQUENCES IN SCHEMA public    TO airis_monolith;
GRANT SELECT         ON ALL TABLES    IN SCHEMA auth      TO airis_monolith;
GRANT SELECT         ON ALL TABLES    IN SCHEMA screening TO airis_monolith;

5. Breaking Changes

Critical: Items 1–4 will immediately break the application if schema moves happen without the compatibility views from Migrations 002/003 being applied first in the same transaction.

#Breaking ChangeSeverityMitigation
1 Cross-schema FK constraints are illegal in PostgreSQL.
ai_screenings.candidate_id → public.candidates, .job_id → public.jobs, .pipeline_id → public.pipelines all become orphaned references once the table moves to the screening schema.		 CRITICAL Migration 003 drops these FKs before SET SCHEMA. Referential integrity is enforced at the application layer: the Screening Service calls GET /api/v1/candidates/{id} and GET /api/v1/jobs/{id} before creating a screening record.
2 All 100+ Alembic migrations reference public.* tables.
After the schema moves, alembic upgrade head on any service fails — it tries to operate on tables that no longer exist in public.		 CRITICAL Each service gets its own alembic.ini with a separate version table (auth.alembic_version, screening.alembic_version). Monolith migrations remain in public.alembic_version. Run alembic stamp head per service after Migration 005.
3 SQLAlchemy models resolve against search_path default (public).
All 13 models being moved will query the wrong schema unless __table_args__ = {"schema": "..."} is added to each class.		 CRITICAL Update all 13 model files (Section 6.1 & 6.2). Compatibility views in public keep the monolith functional while model changes are staged and deployed.
4 get_current_user() in core/dependencies.py makes direct DB queries against Profile (line 165) and AuthSession (line 186).
Every authenticated request across the entire platform depends on these two lookups. If DB access to auth schema is revoked, all endpoints return 503.		 CRITICAL Phase 1A/B: monolith retains read access via compatibility views — no immediate change needed. Phase 1C: replace both lookups with GET {AUTH_URL}/internal/users/{id} and embed permissions as a perms JWT claim to eliminate the DB round-trip on line 79.
5 ai_screening_service.py writes directly to public.pipelines.
The service calls pipeline_service.update_stage() which issues a db.commit() on the Pipeline SQLAlchemy model. After extraction, the Screening Service process has no write access to public.		 HIGH Replace with: PATCH {MONOLITH_URL}/api/v1/pipelines/{id}/stage, authenticated with a shared X-Service-Key header.
6 ai_screening_service.py reads Candidate and Job ORM objects for AI context.
Resume text and job description are fetched via SQLAlchemy join and passed to question_generator.py as ORM model objects.		 HIGH Replace with GET {MONOLITH_URL}/api/v1/candidates/{id} and GET {MONOLITH_URL}/api/v1/jobs/{id}. Update question_generator.py to accept plain dict payloads.
7 invites table is dual-purpose.
Used for recruiter onboarding invites (Auth domain) and AI screening candidate invitations (Screening domain).		 MEDIUM Phase 1: keep in auth schema. Screening Service calls POST {AUTH_URL}/internal/invites to create candidate invite records. Phase 2: add an invite_type discriminator column.
8 Supabase RLS policies may reference public.* table paths explicitly.
Policies on the 13 moved tables will break silently after SET SCHEMA. Compatibility views in public do not inherit RLS from the underlying tables.		 MEDIUM Audit all RLS policies via SELECT * FROM pg_policies WHERE tablename IN (...). Recreate them pointing to the auth.* or screening.* tables.
9 CORS — new service origins.
Auth Service (:8001) and Screening Service (:8002) on different ports or subdomains will trigger browser CORS failures.		 LOW Add CORSMiddleware to each new service with the same FRONTEND_ORIGIN allowed. Or proxy all three services behind a single nginx so the external origin stays identical.

7. Rollout Sequence

The compatibility views are your safety net. Phase 1A physically moves the data while the monolith continues running unchanged, because every moved table has a view of the same name in public. Remove the views only after every cross-schema SQL statement has been replaced by an API call and verified in production.

Phase 1A — Schema Isolation (zero downtime)

1
Back up. Full pg_dump before any schema operations.
2
Run Migration 001 — creates auth and screening schemas. Zero application impact.
3
Run Migration 002 — moves 7 auth tables then immediately creates 7 compatibility views in public. The monolith reads public.profiles transparently (it is now a view over auth.profiles).
4
Run Migration 003 — drops cross-schema FKs, moves 6 screening tables, restores intra-schema FKs within screening, creates 6 compatibility views in public.
5
Run Migration 004 — creates DB roles with schema-scoped permissions.
6
Run Migration 005 — creates auth.alembic_version and screening.alembic_version. Run alembic stamp head in each new service directory to record the current state as the migration baseline.
7
Update SQLAlchemy models — add __table_args__ = {"schema": "..."} to all 13 model classes (Sections 6.1 & 6.2). Deploy updated monolith. Run full smoke test — all routes should pass because the compatibility views remain active.

Phase 1B — Service Extraction (parallel deployment)

8
Bootstrap Auth Service — new auth_service/ FastAPI app with auth routes, 7 auth models (schema=auth), core/security.py, RBAC seeding, and its own Alembic config. DB connection uses role airis_auth. Deploy on port :8001.
9
Bootstrap AI Screening Service — new screening_service/ FastAPI app with AI screening routes, 6 screening models (schema=screening), all AI service files (question_generator, answer_evaluator, recommendation_engine), WebSocket handler, and HTTP clients for Monolith calls. DB role: airis_screening. Deploy on port :8002.
10
Update Monolith — replace the direct pipeline_service call inside ai_screening_service.py with an HTTP PATCH /api/v1/pipelines/{id}/stage call. Set SCREENING_SERVICE_URL and SERVICE_API_KEY env vars.
11
Update Frontend — if Auth/Screening have different origins, set NEXT_PUBLIC_AUTH_URL and NEXT_PUBLIC_SCREENING_URL. Update API client base URLs for auth and screening modules. Add CORS to both new services.

Phase 1C — Cleanup (after validation period)

12
Audit all cross-schema SQL — grep the entire codebase for select(Profile, select(AuthSession, select(AIScreening, role_permissions. Every hit in non-auth/non-screening service code must be replaced by an API call before proceeding.
13
Drop compatibility views — DROP VIEW public.profiles CASCADE for all 13 views. This enforces the service boundary at the DB level: any rogue cross-schema SQL now produces a hard error rather than silently succeeding.
14
Revoke monolith cross-schema access — REVOKE SELECT ON ALL TABLES IN SCHEMA auth FROM airis_monolith. Same for screening. The monolith must now communicate with Auth and Screening exclusively through APIs.
15
(Optional) Embed RBAC in JWT — add a perms: string[] claim to access tokens in the Auth Service. Remove all per-request PermissionService DB lookups from every service. Eliminates the last reason any service needs to read the auth schema directly.
